VOT-INVITADO — Elegibilidad de Votación

📅 18 abril 2026 👤 Para: Ing. Andrés Suárez Soto ✍️ De: Javier Castilla Robles
🔴 Confirmado en prueba — cédula no registrada emitió voto válido
Regla de negocio — no negociable
CÉDULA NO RECONOCIDA = NO VOTA.
No existe flujo "continuar como invitado" dentro del enlace de votación.
Invitados no votan. Sin excepción. Sin bypass.

2 · Modelo de Datos — Campo status
Agregar campo status a members / session_members
// status posibles
status: "habilitado" | "pendiente_validacion" | "invitado"

// regla obligatoria — ejecutar en backend al calcular elegibilidad
if (status === "invitado") eligible_voto = false // siempre. Sin excepción.
if (status === "pendiente_validacion") eligible_voto = false
if (status === "habilitado" && elegible_por_cargo) eligible_voto = true
3 · Arquitectura — Dónde vive el bloqueo
⚙️ Backend — Fuente de verdad
  • Endpoint POST /api/voting/validate recibe cedula + voting_id → devuelve { eligible: bool, status, cargo }
  • Si eligible = false → devuelve HTTP 403. El frontend no muestra el formulario.
  • El endpoint POST /api/voting/submit revalida elegibilidad antes de registrar. Si no pasa → HTTP 403. Nunca confiar en el frontend.
  • Validación en middleware del endpoint de submit: si eligible_voto !== true → rechazar antes de tocar la lógica de voto.
🖥️ Frontend — Solo refleja lo que dice el backend
  • Al recibir cédula → llamar /api/voting/validate antes de renderizar cualquier opción.
  • Si backend devuelve eligible: false → mostrar mensaje de bloqueo. No renderizar el formulario.
  • El botón "Confirmar Voto" solo existe en DOM si eligible: true fue devuelto por el backend.
  • Eliminar completamente el botón / lógica "Continuar como invitado" del componente de votación.
🔒 Seguridad — Prevenir bypass
  • Acceso directo por URL (/public/voting/16): el backend valida cédula antes de servir cualquier dato. Sin token válido → pantalla de ingreso de cédula. Sin cédula elegible → bloqueo.
  • Manipulación del enlace: el voting_id en la URL no otorga elegibilidad — la elegibilidad viene de la BD, no del enlace.
  • Doble validación obligatoria: frontend valida para UX, backend valida para seguridad. Si el frontend falla o es manipulado, el backend rechaza igual.
4 · UI — Mensaje de bloqueo
🚫
No es posible registrar su voto
La cédula ingresada no se encuentra habilitada para esta votación. Solo pueden votar personas registradas en la reunión con derecho a voto.
Botón único: "Reintentar" · Sin opción "Continuar como invitado" · Sin acceso al formulario

6 · Impacto en Resultados · PDF · Dashboard
Regla de integridad — sin excepción
  • Votos de cédulas no elegibles nunca entran al conteo. Si llegaron por el bug actual, deben invalidarse retroactivamente en las sesiones de prueba.
  • La tabla de votos debe incluir member_id + eligible_voto_al_momento para trazabilidad. Si eligible_voto = false al momento del voto → voto marcado como inválido.
  • El PDF de resultados y el dashboard solo agregan votos donde eligible_voto = true. Nunca conteo bruto de la tabla de votos.

Implementar en este orden exacto
  • 1. Backend primero: Agregar campo status a members. Implementar middleware de elegibilidad en /api/voting/submit. Revalidar aunque el frontend diga que es elegible.
  • 2. Eliminar del frontend: Borrar botón/lógica "Continuar como invitado" del componente de votación. Reemplazar por mensaje de bloqueo con botón "Reintentar".
  • 3. Correr los 6 tests antes de cerrar el fix. Verificar especialmente TEST-04 (bypass URL) y TEST-05 (doble submit) — son los más frecuentes en producción.
